# InvertedPendulum Survival Scoreboard World Model Homework

This homework evaluates a learned dynamics model directly, without policy
training, reward prediction, actor-critic updates, or custom MuJoCo XML. The
environment is Gymnasium's official `InvertedPendulum-v5`.

Students receive real MuJoCo state/action windows and train one world model:

```text
s_hat[t+1] = f_theta(s_hat[t], a[t])
```

The model sees a short ground-truth warmup, then predicts autoregressively until
it fails or reaches the configured scoreboard horizon. The main score is **H80**:
the largest prediction horizon at which at least 80% of test windows have not
failed.

## Why This Assignment

World models are often used to help policies, but policy return can hide whether
the model actually predicts physical dynamics. Here MuJoCo is the ground-truth
physics simulator. The learned model is judged by how long it can stay close to
MuJoCo under the same action sequence.

```text
MuJoCo trajectory:      s_0:T, a_0:T-1
Learned world model:   warmup -> open-loop predictions
Official metric:       survival horizon scoreboard
```

## Assignment Boundary

Students edit only:

```text
student/model.py
student/rollout.py
student/losses.py
student/metrics.py
configs/student.yaml
```

Students do not edit locked data generation, official evaluation, hidden seeds,
failure thresholds, or the baseline model. There is no controller submission.
Action sequences are generated by locked staff code using a fixed stabilizing
feedback term plus smooth noise.

## Data And Scoreboard Horizon

The canonical dataset fields are:

```text
warmup_steps: 5
max_horizon: configurable, e.g. 100 for Colab, 1000 for scoreboard
states:  [N, warmup_steps + max_horizon + 1, 4]
actions: [N, warmup_steps + max_horizon, 1]
```

State order:

```text
[cart position x, pole angle theta, cart velocity x_dot, pole angular velocity theta_dot]
```

With `warmup_steps=5`, the model may consume true states `s_0..s_4`, starts
open-loop from `s_5`, and then uses only its own predicted states. Only finite
windows with `abs(theta_true) < 0.20` for the full window are kept.

## Quick Start

```bash
python -m pip install -r requirements.txt
pytest -q

python -m wm_hw.dataset --config configs/colab.yaml --output-dir data/public --smoke
python -m wm_hw.train --config configs/baseline.yaml --model baseline --dataset-dir data/public --output-dir artifacts/baseline --smoke

python -m wm_hw.eval_horizon --checkpoint-dir artifacts/student/best_checkpoint --dataset-dir data/public --split test --horizon auto --output-dir artifacts/student/eval_test

python -m wm_hw.plotting --eval-dir artifacts/student/eval_test --output-dir artifacts/student/plots
```

## Scoreboard Run

The regular Colab config stays lightweight. For a longer leaderboard-style run,
generate longer windows and evaluate with `--horizon auto`:

```bash
python -m wm_hw.dataset --config configs/scoreboard.yaml --output-dir data/scoreboard
python -m wm_hw.eval_horizon --checkpoint-dir artifacts/student/best_checkpoint --dataset-dir data/scoreboard --split test --horizon auto --output-dir artifacts/student/scoreboard_test
```

`configs/scoreboard.yaml` defaults to `max_horizon: 1000`. Teachers may change
that value to 500, 2000, or another horizon without editing code.

## Metrics

Primary metric:

```text
H80 = max h such that at least 80% of windows survive to step h
```

Failure requires two consecutive violated prediction steps. Default thresholds:

```text
angle_error_rad = 0.075
cart_pos_error_m = 0.10
cart_vel_error_mps = 0.75
pole_vel_error_radps = 1.00
consecutive_fail_steps = 2
```

Official evaluation writes:

```text
metrics.json
scoreboard_summary.json
per_window_survival.npy
```

Important metrics:

```text
max_horizon
H80
H50
survival_auc
mean_survival_steps
median_survival_steps
success_rate@configured_milestones
one_step_rmse
open_loop_rmse@horizon
```
